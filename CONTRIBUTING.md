# 기여 가이드라인

to-tech-lead 프로젝트에 기여해 주셔서 감사합니다!

이 문서는 프로젝트에 기여하는 방법을 안내합니다.

## 기여 방법

### 1. 이슈 등록

새로운 아이디어, 버그 리포트, 개선 제안이 있으시면 [Issues](https://github.com/nathankim0/to-tech-lead/issues)에 등록해 주세요.

- **버그 리포트**: 잘못된 링크, 오타, 내용 오류 등
- **기능 요청**: 새로운 학습 리소스, 주제, 섹션 추가 제안
- **개선 제안**: 기존 내용의 보완, 구조 개선 등

### 2. Pull Request

#### 시작하기 전에

1. 먼저 관련 이슈가 있는지 확인해 주세요
2. 없다면 이슈를 먼저 등록하고 논의해 주세요
3. 작업을 시작하기 전에 `main` 브랜치에서 새 브랜치를 생성하세요

#### 브랜치 네이밍 규칙

```
feat/주제-설명     # 새로운 기능/콘텐츠 추가
fix/주제-설명      # 버그/오류 수정
docs/주제-설명     # 문서 개선
refactor/주제-설명 # 구조 리팩토링
```

#### 커밋 메시지 규칙

```
feat: 새로운 기능/콘텐츠 추가
fix: 버그/오류 수정
docs: 문서 수정
style: 포맷팅, 세미콜론 등 (코드 변경 없음)
refactor: 코드 리팩토링
chore: 빌드, 설정 파일 수정
```

예시:
```
feat: Kubernetes 학습 리소스 추가
fix: 잘못된 링크 수정
docs: README 구조 개선
```

### 3. 콘텐츠 기여 가이드

#### 새로운 학습 리소스 추가 시

- 무료로 접근 가능한 리소스를 우선적으로 추가해 주세요
- 한국어 리소스가 있다면 함께 추가해 주세요
- 리소스의 난이도와 예상 학습 시간을 명시해 주세요
- 실제로 학습해 본 리소스를 권장합니다

#### 마크다운 작성 규칙

- 헤딩은 `#`부터 순차적으로 사용합니다
- 링크는 설명과 함께 작성합니다: `[설명](URL)`
- 코드 블록에는 언어를 명시합니다
- 이모지는 적절히 사용하되 과도하게 사용하지 않습니다

#### 링크 규칙

- 모든 외부 링크는 HTTPS를 사용합니다
- 깨진 링크가 없는지 확인해 주세요
- 가능하면 공식 문서 링크를 사용합니다

## 리뷰 프로세스

1. PR을 제출하면 자동으로 링크 검사가 실행됩니다
2. 메인테이너가 리뷰를 진행합니다
3. 필요한 경우 수정 요청이 있을 수 있습니다
4. 승인되면 `main` 브랜치에 머지됩니다

## 행동 강령

- 모든 참여자를 존중해 주세요
- 건설적인 피드백을 제공해 주세요
- 다양한 의견과 경험을 환영합니다
- 혐오 발언, 차별, 괴롭힘은 용납되지 않습니다
