# LEVEL 6 - 테크리드 (Tech Lead)

## 개요

테크리드는 기술적 의사결정과 팀 리딩을 동시에 수행하는 역할입니다. 단순히 코드를 잘 짜는 것을 넘어, 팀 전체의 기술 방향성을 설정하고, 팀원들이 성장할 수 있는 환경을 만들어야 합니다.

> "테크리드는 기술로 문제를 해결하는 사람이 아니라, 팀이 기술로 문제를 해결할 수 있게 만드는 사람이다."

---

## 학습 목표

이 레벨을 완료하면 다음 역량을 갖추게 됩니다:

### 핵심 역량
- **기술 의사결정 가능**: 아키텍처, 기술 스택, 라이브러리 선택 등 주요 기술 결정을 내리고 그 근거를 명확히 설명할 수 있다
- **팀 설계 가능**: 프로젝트 특성에 맞는 팀 구조를 설계하고, 효과적인 협업 프로세스를 구축할 수 있다

### 세부 역량
| 역량 | 설명 | 숙련도 목표 |
|------|------|------------|
| 기술 의사결정 | ADR 작성, 기술 로드맵 수립, 트레이드오프 분석 | ★★★★☆ |
| 팀 구조 설계 | 스쿼드/길드, 역할 정의, 온보딩 설계 | ★★★★☆ |
| 코드리뷰 문화 | 리뷰 가이드라인, 지식 공유 체계, 품질 기준 | ★★★★★ |
| 기술 부채 관리 | 부채 식별, 우선순위화, 리팩토링 전략 | ★★★★☆ |

## AI 학습 프롬프트

테크리드 역량을 키우기 위해 AI를 코치로 활용할 수 있습니다.

### 테크리드 코치 프롬프트

```
당신은 10년 경력의 시니어 테크리드입니다. 모바일 개발팀을 이끌며
다양한 기술 의사결정과 팀 빌딩 경험이 있습니다.

저는 시니어 모바일 개발자에서 테크리드로 성장하려고 합니다.
다음 원칙에 따라 코칭해주세요:

1. 정답을 알려주기보다 질문으로 생각을 유도해주세요
2. 실제 경험담과 사례를 공유해주세요
3. 실패 경험과 그로부터 배운 점도 알려주세요
4. 모바일 개발 맥락에서 조언해주세요

현재 상황: [구체적인 상황 설명]
고민: [의사결정이 필요한 내용]
```

### 상황별 프롬프트 예시

#### 기술 의사결정이 필요할 때
```
테크리드로서 다음 기술 선택을 해야 합니다.
- 상황: [예: 새로운 상태관리 라이브러리 도입 검토]
- 후보 A: [예: Redux]
- 후보 B: [예: Zustand]
- 고려사항: [팀 규모, 학습 곡선, 유지보수 등]

ADR(Architecture Decision Record) 형식으로
의사결정 과정을 정리하는 것을 도와주세요.
```

#### 팀 갈등 상황
```
테크리드로서 팀 내 기술적 의견 충돌을 중재해야 합니다.
- 상황: [예: 코드 스타일 통일 논쟁]
- A 의견: [상세 내용]
- B 의견: [상세 내용]

어떻게 합의점을 찾고 팀을 이끌어야 할까요?
```

#### 기술 부채 논의
```
PO/PM에게 기술 부채 해결의 필요성을 설득해야 합니다.
- 현재 부채: [예: 레거시 API 계층]
- 영향: [예: 신규 기능 개발 속도 30% 저하]
- 제안: [예: 2주간 리팩토링 스프린트]

비개발자도 이해할 수 있게 설명하는 법을 코칭해주세요.
```

## 산출물

이 레벨을 완료하면 다음 산출물을 만들 수 있어야 합니다.

### 1. 기술 의사결정 템플릿 (ADR)

```markdown
# ADR-XXX: [의사결정 제목]

## 상태
[제안됨 | 승인됨 | 폐기됨 | 대체됨]

## 컨텍스트
- 현재 상황과 문제점
- 왜 이 결정이 필요한가?

## 결정
- 무엇을 결정했는가?
- 선택한 옵션은?

## 근거
- 왜 이 결정을 내렸는가?
- 고려한 대안들과 각각의 장단점
- 트레이드오프 분석

## 결과
- 이 결정으로 인한 예상 결과
- 필요한 후속 작업
- 성공 지표

## 참여자
- 의사결정 참여자 목록
- 검토일: YYYY-MM-DD
```

### 2. 팀 운영 가이드

```markdown
# [팀명] 운영 가이드

## 팀 미션
[팀의 존재 이유와 목표]

## 팀 구조
[역할, 책임, 의사결정 권한]

## 일하는 방식
### 회의
- 스탠드업: [시간, 형식]
- 스프린트 플래닝: [주기, 형식]
- 회고: [주기, 형식]

### 코드리뷰
- 필수 리뷰어 수: N명
- 응답 SLA: N시간 이내
- 리뷰 체크리스트: [링크]

### 기술 부채 관리
- 부채 트래킹: [도구, 방법]
- 리팩토링 시간: 스프린트의 N%

## 온보딩
- 1주차: [내용]
- 2주차: [내용]
- 첫 달 목표: [내용]

## 팀 문화
- 핵심 가치 1
- 핵심 가치 2
- 핵심 가치 3
```

## 자가 체크리스트

### 기술 의사결정
- [ ] ADR을 작성하고 팀과 공유할 수 있다
- [ ] 기술 선택의 트레이드오프를 명확히 설명할 수 있다
- [ ] 비개발 직군에게 기술 결정의 영향을 설명할 수 있다
- [ ] 기술 로드맵을 수립하고 관리할 수 있다

### 팀 구조 설계
- [ ] 프로젝트 특성에 맞는 팀 구조를 제안할 수 있다
- [ ] 역할과 책임(R&R)을 명확히 정의할 수 있다
- [ ] 효과적인 온보딩 프로세스를 설계할 수 있다
- [ ] 팀 간 협업 방식을 설계할 수 있다

### 코드리뷰 문화
- [ ] 팀의 코드리뷰 가이드라인을 작성할 수 있다
- [ ] 건설적인 피드백을 주고받는 문화를 만들 수 있다
- [ ] 코드리뷰를 통한 지식 공유 체계를 구축할 수 있다

### 기술 부채 관리
- [ ] 기술 부채를 식별하고 분류할 수 있다
- [ ] 부채 해결의 우선순위를 정할 수 있다
- [ ] 리팩토링 전략을 수립하고 실행할 수 있다
- [ ] 부채 관리의 필요성을 이해관계자에게 설득할 수 있다

## 성장 단계별 체크리스트

테크리드로 성장하는 과정을 3단계로 나누어 체크리스트를 제공합니다.

### Stage 1: 초기 테크리드 (0-6개월)

**이 단계의 목표**: 팀의 신뢰를 얻고, 기본적인 리더십 활동을 시작한다.

#### 기술 영역
- [ ] 팀의 코드베이스를 충분히 이해했다
- [ ] 주요 아키텍처 결정의 배경을 파악했다
- [ ] 팀의 기술 스택과 도구를 능숙하게 다룬다
- [ ] 기술 부채 목록을 작성했다
- [ ] 첫 번째 ADR을 작성했다

#### 리더십 영역
- [ ] 모든 팀원과 1:1을 시작했다
- [ ] 팀원들의 강점과 성장 포인트를 파악했다
- [ ] 코드리뷰 가이드라인을 팀과 함께 정의했다
- [ ] 스탠드업/회고를 효과적으로 진행한다
- [ ] PM/기획자와 정기적으로 소통한다

#### 소통 영역
- [ ] 팀 외부(PM, 디자이너, 다른 팀)와 원활히 소통한다
- [ ] 기술적 결정을 비개발자에게 설명할 수 있다
- [ ] 회의에서 팀을 대표하여 의견을 전달한다
- [ ] 문서화 습관을 들였다

#### 이 단계에서 흔히 하는 실수
- 여전히 개인 기여자처럼 모든 코드를 직접 작성하려 함
- 팀원에게 위임하지 않고 혼자 문제를 해결하려 함
- 피드백을 주는 것을 불편해 함
- 회의 시간이 급격히 늘어나는 것에 당황함

### Stage 2: 성장 테크리드 (6개월-2년)

**이 단계의 목표**: 팀의 기술 방향을 주도하고, 팀원들의 성장을 이끈다.

#### 기술 영역
- [ ] 분기별 기술 로드맵을 수립한다
- [ ] 기술 부채 해결에 정기적으로 시간을 투자한다
- [ ] 팀의 코드 품질이 향상되고 있다 (메트릭으로 확인)
- [ ] 새로운 기술 도입 시 ADR로 결정 과정을 문서화한다
- [ ] 장애 발생 시 팀을 이끌고 해결한다

#### 리더십 영역
- [ ] 팀원들에게 적절한 수준의 도전적인 업무를 배분한다
- [ ] 주니어 개발자를 효과적으로 멘토링한다
- [ ] 팀 내 갈등을 중재할 수 있다
- [ ] 채용 인터뷰에 참여하고 좋은 인재를 판별한다
- [ ] 팀의 온보딩 프로세스를 개선했다

#### 소통 영역
- [ ] 경영진에게 기술 현황을 보고한다
- [ ] 기술 부채 해결의 필요성을 설득할 수 있다
- [ ] 크로스 팀 협업을 원활하게 이끈다
- [ ] 기술 공유 세션을 정기적으로 진행한다

#### 이 단계에서 흔히 하는 실수
- 직접 코딩하는 시간이 너무 줄어들어 기술력이 떨어짐
- 모든 의사결정을 혼자 하려고 함
- 팀원의 성장보다 업무 완료에만 집중함
- 번아웃 조짐을 무시함

### Stage 3: 숙련 테크리드 (2년 이상)

**이 단계의 목표**: 팀을 넘어 조직의 기술 방향에 영향을 미친다.

#### 기술 영역
- [ ] 조직 전체의 기술 표준 수립에 기여한다
- [ ] 다른 팀의 기술 리뷰에 참여한다
- [ ] 장기적인(1년 이상) 기술 비전을 제시한다
- [ ] 대규모 마이그레이션/리팩토링을 성공적으로 이끌었다
- [ ] 기술 선택이 비즈니스에 미치는 영향을 정량화한다

#### 리더십 영역
- [ ] 후임 테크리드를 양성한다
- [ ] 팀의 자율성을 높여 본인 없이도 팀이 잘 돌아간다
- [ ] 조직 문화에 긍정적인 영향을 미친다
- [ ] 어려운 대화(저성과자 피드백, 해고 등)를 할 수 있다
- [ ] 팀원들이 테크리드가 되도록 성장시켰다

#### 소통 영역
- [ ] 회사 외부(컨퍼런스, 블로그 등)에서 기술 리더십을 발휘한다
- [ ] 채용 브랜딩에 기여한다
- [ ] 경영진의 기술 전략 수립에 참여한다
- [ ] 업계 네트워크를 구축했다

#### 다음 단계 고려
이 수준에 도달하면 다음 경로를 고려할 시점입니다:
- **엔지니어링 매니저**: 피플 매니지먼트에 집중
- **스태프 엔지니어**: 깊은 기술 전문성 추구
- **아키텍트**: 시스템 설계 전문가로 발전
