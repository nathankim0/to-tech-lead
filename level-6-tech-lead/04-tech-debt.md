# 기술 부채 관리

## 1. 한 줄 요약

**기술 부채는 "나중에 갚아야 할 개발 비용"이며, 테크리드의 핵심 역할은 부채를 0으로 만드는 것이 아니라 적정 수준으로 관리하는 것이다.**

---

## 2. 쉽게 설명

### 모바일 개발자 관점 비유

**기술 부채는 신용카드 빚과 같습니다.**

- 신용카드로 물건을 사면 당장은 편리하지만, 나중에 이자와 함께 갚아야 합니다
- 적당히 사용하면 현금 흐름 관리에 도움이 되지만, 과도하면 파산합니다
- 부채가 있다고 무조건 나쁜 것이 아니라, **통제 가능한 수준**인지가 중요합니다

코드에서 "일단 이렇게 해놓고 나중에 고치자"라고 할 때마다 부채가 쌓입니다.
- 당장 빠르게 출시할 수 있지만 (편리함)
- 나중에 수정할 때 더 많은 시간이 듭니다 (이자)
- 너무 쌓이면 새 기능 개발이 불가능해집니다 (파산)

### 기술 부채의 유형

| 유형 | 설명 | 모바일 예시 |
|------|------|------------|
| **의도적 부채** | 알면서 선택한 부채 | "출시 일정 맞추려고 테스트 없이 배포" |
| **비의도적 부채** | 모르고 만든 부채 | "처음엔 좋은 설계였는데 요구사항 변경으로 복잡해짐" |
| **비트 부패(Bit Rot)** | 시간이 지나 낡아진 것 | "3년 전 라이브러리, 더 이상 유지보수 안 됨" |
| **컨텍스트 부채** | 지식 손실 | "이 코드 왜 이런지 아는 사람이 없음" |

### 테크리드의 역할

주니어 때는 부채를 **만들거나** **발견**했다면, 테크리드는:
- 부채를 **식별**하고 **분류**한다
- **우선순위**를 정한다
- **이해관계자를 설득**하여 해결 시간을 확보한다
- **점진적 개선 전략**을 수립한다

## 4. 실무 적용 예시

### 예시 1: 기술 부채 인벤토리

```markdown
# 기술 부채 인벤토리

## 아키텍처

### DEBT-001: 레거시 네트워크 계층
- **상태**: 계획됨
- **우선순위**: 높음
- **영향**: 새 API 연동 시 2-3일 추가 작업
- **추정 비용**: 3주
- **이해관계자**: 전체 개발팀

**설명**
현재 네트워크 계층이 Retrofit + RxJava 기반으로 되어있음.
새로운 기능은 Coroutine + Flow 기반으로 개발하고 있어
두 패러다임이 혼재되어 있음.

**영향 분석**
- 신규 개발자 온보딩 시간 증가 (+1주)
- 테스트 작성 복잡도 증가
- 에러 핸들링 일관성 부족

**해결 방안**
1. 새 NetworkClient 인터페이스 정의
2. 점진적 마이그레이션 (기능별)
3. 레거시 계층 deprecation

### DEBT-002: God Object - UserManager
- **상태**: 백로그
- **우선순위**: 중간
- **영향**: 변경 시 사이드 이펙트 발생 가능
- **추정 비용**: 2주
- **이해관계자**: 인증팀

**설명**
UserManager 클래스가 1500줄, 30개 이상의 메서드를 가짐.
인증, 프로필, 설정, 알림 등 여러 책임이 혼재.

**해결 방안**
1. 도메인별로 분리 (AuthManager, ProfileManager, ...)
2. 페사드 패턴으로 하위 호환성 유지
3. 단계적 마이그레이션

## 테스트

### DEBT-003: UI 테스트 부재
- **상태**: 진행 중
- **우선순위**: 높음
- **영향**: 리그레션 버그 발생 위험
- **추정 비용**: 지속적 (스프린트당 10%)
- **이해관계자**: QA팀, 개발팀

**현황**
- 단위 테스트: 60% 커버리지
- 통합 테스트: 30% 커버리지
- UI 테스트: 5% 커버리지

**해결 방안**
1. 핵심 사용자 여정 선정 (5개)
2. 스프린트당 1개씩 E2E 테스트 추가
3. CI에 통합

## 의존성

### DEBT-004: 구버전 라이브러리
- **상태**: 계획됨
- **우선순위**: 중간
- **영향**: 보안 취약점, 새 기능 사용 불가
- **추정 비용**: 1주

**목록**
| 라이브러리 | 현재 버전 | 최신 버전 | 위험도 |
|-----------|----------|----------|--------|
| OkHttp | 3.14 | 4.12 | 중 |
| Glide | 4.11 | 4.16 | 낮 |
| Room | 2.4 | 2.6 | 낮 |
```

### 예시 2: 이해관계자 설득 자료

**상황**: PM에게 기술 부채 해결 시간을 요청해야 함

```markdown
# 기술 부채 해결 제안서

## 요약
다음 분기에 스프린트의 20%를 기술 부채 해결에 투자하고 싶습니다.
이를 통해 장기적으로 개발 속도를 30% 향상시킬 수 있습니다.

## 현재 상황

### 정량적 데이터
| 지표 | 현재 | 1년 전 | 변화 |
|------|------|--------|------|
| 평균 기능 개발 시간 | 5일 | 3일 | +67% |
| 버그 발생률 | 주 8건 | 주 3건 | +167% |
| 빌드 시간 | 12분 | 5분 | +140% |
| 새 개발자 온보딩 | 4주 | 2주 | +100% |

### 비용으로 환산
- 개발 시간 증가: 월 40시간 × 5명 = **월 200시간 손실**
- 버그 수정 시간: 주 16시간 = **월 64시간 손실**
- 총: **월 264시간** (약 1.5 FTE)

## 제안

### 투자
- 기간: 다음 분기 (12주)
- 비용: 스프린트의 20% = **주 2일 × 12주 = 24일**

### 기대 효과
| 목표 | 현재 | 목표 | 개선 |
|------|------|------|------|
| 기능 개발 시간 | 5일 | 3.5일 | -30% |
| 버그 발생률 | 주 8건 | 주 4건 | -50% |
| 빌드 시간 | 12분 | 6분 | -50% |

### ROI 계산
- 투자: 24일 (1.2 FTE-월)
- 기대 절감: 월 200시간 → 월 140시간 = **월 60시간 절감**
- 손익분기: **3개월** 후
- 1년 기준 절감: **720시간 (약 4 FTE-월)**

## 구체적 계획

### 1순위: 네트워크 계층 리팩토링 (2주)
- 영향: 새 API 개발 속도 2배
- 담당: 김개발

### 2순위: 빌드 시스템 개선 (1주)
- 영향: 빌드 시간 50% 감소
- 담당: 이빌드

### 3순위: 테스트 커버리지 (지속)
- 영향: 버그 발생률 50% 감소
- 담당: 전체 팀

## 결론
**지금 투자하지 않으면**, 6개월 후에는 기능 개발 시간이 7일로 늘어나고,
버그 발생률은 주 12건으로 증가할 것으로 예상됩니다.

현명한 투자로 기술적 역량을 회복하고,
더 빠른 가치 전달이 가능한 팀이 되겠습니다.
```

## 7. 추가 질문

### 기본 이해 확인

**1. 의도적 기술 부채와 비의도적 기술 부채의 차이점은?**

> **답변**: 의도적 기술 부채는 트레이드오프를 인지한 상태에서 전략적으로 선택한 부채입니다. "출시 일정을 맞추기 위해 테스트 커버리지를 50%만 확보하고 가겠다"처럼 비용과 이익을 저울질한 결과입니다. 반면 비의도적 기술 부채는 지식 부족이나 예상치 못한 요구사항 변경으로 인해 발생합니다. "처음엔 좋은 설계였는데 요구사항이 바뀌면서 복잡해졌다"가 대표적입니다. 테크리드 관점에서 의도적 부채는 언제, 어떻게 갚을지 계획이 있어야 하고, 비의도적 부채는 정기적인 코드 리뷰와 회고를 통해 발견하고 관리해야 합니다. 가장 위험한 것은 의도적이지만 무모한 부채입니다. "설계할 시간이 없어서 그냥 했다"는 발견조차 어렵고, 이자가 기하급수적으로 늘어납니다.

**2. 기술 부채가 0이 되는 것이 목표가 아닌 이유는?**

> **답변**: 기술 부채 0을 목표로 하면 오히려 비즈니스에 해가 됩니다. 첫째, 기회비용의 문제입니다. 완벽한 코드를 만드는 데 투자하는 시간에 경쟁사가 시장을 선점할 수 있습니다. 둘째, 과잉 설계(Over-engineering)의 위험입니다. 미래의 모든 요구사항을 예측해서 설계하려다 오히려 복잡하고 유지보수 어려운 코드가 됩니다. 셋째, 적정 수준의 부채는 건강합니다. 신용카드처럼 현금 흐름(개발 속도)을 관리하는 도구로 활용할 수 있습니다. 테크리드의 역할은 부채를 0으로 만드는 것이 아니라, 팀이 감당 가능한 수준으로 유지하면서 언제든 갚을 수 있는 계획을 세우는 것입니다. 일반적으로 개발 시간의 10-20%를 부채 관리에 투자하는 것이 적정선입니다.

**3. 기술 부채의 "이자"란 무엇인가?**

> **답변**: 기술 부채의 이자는 부채를 갚지 않고 유지할 때 지불하는 추가 비용입니다. 구체적으로 살펴보면: 1) **개발 시간 증가** - 깨끗한 코드라면 1일 걸릴 작업이 레거시 코드에서는 3일이 걸립니다. 2) **버그 발생률 증가** - 복잡한 코드는 변경 시 예상치 못한 사이드 이펙트를 발생시킵니다. 3) **온보딩 비용 증가** - 새로운 팀원이 코드를 이해하는 데 더 오래 걸립니다. 4) **팀 사기 저하** - 매일 레거시 코드와 싸우면 개발자들이 지칩니다. 5) **기술 인재 이탈** - 좋은 개발자들은 기술 부채가 많은 프로젝트를 떠납니다. 무서운 점은 이자가 복리로 쌓인다는 것입니다. 부채 위에 부채가 쌓이면 이자율이 기하급수적으로 증가하여 결국 "기술 파산" 상태가 됩니다.

### 심화 질문

**4. 비개발자에게 기술 부채를 어떻게 설명할 것인가?**

> **답변**: 비개발자에게는 "집"이나 "신용카드" 비유가 효과적입니다. "우리 코드는 10년 된 집과 같습니다. 처음엔 깨끗했지만, 급하게 방을 늘리고, 배관을 임시로 연결하고, 페인트 대신 테이프로 때웠습니다. 이제 새 방을 만들려면 기존 구조물을 피해서 작업해야 하고, 가끔 예전에 임시로 한 것들이 터집니다." 또는 숫자로 설명합니다: "작년에는 이 기능을 3일에 만들었는데, 지금은 같은 크기의 기능에 7일이 걸립니다. 이 차이가 기술 부채의 이자입니다." PM에게는 ROI로, 경영진에게는 리스크와 기회비용으로 설명합니다. "지금 2주를 투자하면, 향후 1년간 매달 1주를 절약할 수 있습니다. 투자하지 않으면 6개월 후에는 신규 기능 개발이 사실상 불가능해집니다."

**5. 기술 부채 해결과 신규 기능 개발의 적절한 비율은?**

> **답변**: 일반적인 가이드라인은 스프린트의 15-20%를 기술 부채 해결에 투자하는 것입니다. 하지만 상황에 따라 조정이 필요합니다. **초기 스타트업(제품-시장 적합성 검증 중)**: 5-10%. 속도가 우선이고, 피봇할 수 있으므로 부채에 너무 집착하지 않습니다. **성장기 스타트업**: 15-20%. 확장을 위해 기술 기반이 중요해집니다. **성숙한 제품**: 20-25%. 안정성과 유지보수성이 핵심입니다. **위기 상황(기술 파산 임박)**: 30-50%까지 일시적으로 증가. 실제로는 "부채 예산"을 정해두고 매 스프린트마다 의무적으로 사용하는 것이 좋습니다. "시간 나면 하자"는 절대 안 됩니다. Google은 "20% 프로젝트"를 통해, Atlassian은 "ShipIt Day"를 통해 정기적으로 부채를 해결합니다.

**6. 기술 부채를 정량적으로 측정하는 방법은?**

> **답변**: 기술 부채를 직접 측정하기는 어렵지만, 다양한 프록시 지표를 활용할 수 있습니다. **코드 메트릭**: Cyclomatic Complexity(순환 복잡도), 코드 중복률, 테스트 커버리지. SonarQube 같은 도구가 "기술 부채"를 시간 단위로 추정해줍니다. **개발 속도 지표**: 기능 개발 리드타임 변화, 버그 수정에 걸리는 시간, PR 리뷰 시간. 이 지표들이 증가하면 부채가 쌓이고 있다는 신호입니다. **팀 경험 지표**: 개발자 설문(코드베이스 건강도 1-10점), 특정 영역 작업 기피 정도, 온보딩 기간. **비용 환산**: (현재 개발 시간 - 이상적 개발 시간) × 인건비로 환산합니다. "이 부채 때문에 매달 200시간을 추가로 쓰고 있으며, 이는 연간 3억 원의 비용입니다"라고 보고하면 경영진도 이해합니다.

### 실습 과제

**7. 현재 프로젝트의 기술 부채 인벤토리를 작성해보자.**

> **답변**: 기술 부채 인벤토리 작성 방법을 안내합니다. 1) **수집 단계**: 팀원들에게 "코드베이스에서 가장 고치고 싶은 것"을 물어봅니다. 코드 리뷰에서 반복적으로 지적된 사항, 회고에서 나온 기술 이슈, 정적 분석 도구 결과를 수집합니다. 2) **분류 단계**: 아키텍처, 테스트, 의존성, 문서, 성능, 보안으로 카테고리를 나눕니다. 3) **평가 단계**: 각 항목에 대해 영향도(1-5), 해결 비용(시간), 긴급도(1-5)를 평가합니다. 4) **문서화 단계**: 부채 ID, 제목, 설명, 영향, 예상 비용, 담당자, 상태를 포함한 표를 만듭니다. 5) **유지 단계**: 스프린트 회고마다 새 부채를 추가하고, 해결된 부채를 정리합니다. Jira, Notion, 또는 단순 스프레드시트로 관리할 수 있습니다.

**8. 가장 큰 기술 부채에 대한 해결 제안서를 작성해보자.**

> **답변**: 해결 제안서 작성 프레임워크입니다. 1) **요약 (1문단)**: "X 문제를 해결하기 위해 Y를 제안하며, 이를 통해 Z 효과를 기대합니다." 2) **현재 상황 (정량적 데이터)**: 버그 발생률, 개발 시간 증가율, 영향받는 기능 수를 숫자로 제시합니다. 3) **비용 분석**: 현재 상태 유지 비용(월간 추가 개발 시간, 버그 수정 비용) vs 해결 비용(투자 시간, 인력)을 비교합니다. 4) **제안 솔루션**: 구체적인 기술 접근 방식, 단계별 계획, 마일스톤을 제시합니다. 5) **위험 분석**: 해결 과정의 리스크와 완화 방안을 설명합니다. 6) **ROI 계산**: "3주 투자 → 매월 40시간 절감 → 3개월 후 손익분기, 1년 후 400시간 순이익"처럼 명확히 보여줍니다. 7) **다음 단계**: 승인 후 즉시 실행할 수 있는 액션 아이템을 제시합니다.

**9. 기술 부채 해결의 ROI를 계산해보자.**

> **답변**: ROI 계산 실습 가이드입니다. **1단계 - 현재 비용 측정**: 최근 3개월 데이터를 수집합니다. 평균 기능 개발 시간, 버그 수정에 소요되는 시간, 코드 이해에 걸리는 시간을 측정합니다. **2단계 - 개선 후 예상 비용**: 유사한 "깨끗한" 영역의 작업 시간을 참고하거나, 업계 벤치마크를 활용합니다. 보통 30-50% 개선을 목표로 합니다. **3단계 - 투자 비용**: 리팩토링에 필요한 인력 × 기간을 계산합니다. 테스트 작성, 마이그레이션, 문서화 시간도 포함합니다. **4단계 - ROI 공식**: `(연간 절감액 - 투자 비용) / 투자 비용 × 100%`. 예: 투자 120시간, 월간 절감 40시간이면 연간 480시간 절감, ROI = (480-120)/120 × 100% = 300%. **5단계 - 손익분기점**: 투자 비용 / 월간 절감액. 위 예시에서는 3개월.

### 토론 주제

**10. "기술 부채를 지는 것"과 "나쁜 코드를 작성하는 것"의 차이는?**

> **답변**: 핵심 차이는 "의도성"과 "상환 계획"입니다. 기술 부채를 진다는 것은: 1) 트레이드오프를 인지하고 있습니다 - "이렇게 하면 나중에 X를 해야 한다는 것을 안다". 2) 비즈니스 가치를 위한 선택입니다 - "지금 출시해서 시장을 선점하는 것이 더 중요하다". 3) 상환 계획이 있습니다 - "출시 후 2주 내에 리팩토링한다". 4) 기록합니다 - "DEBT-XXX로 백로그에 등록". 반면 나쁜 코드는: 1) 더 나은 방법을 모르거나 신경 쓰지 않습니다. 2) 상환 계획 없이 "나중에 고치자"만 반복합니다. 3) 문서화하지 않습니다. 테크리드는 팀이 "의도적이고 신중한 부채"만 지도록 가이드해야 합니다. 모든 부채에는 만기일과 담당자가 있어야 합니다.

**11. 스타트업 초기에 기술 부채는 필요악인가?**

> **답변**: 네, 초기 스타트업에서 어느 정도의 기술 부채는 생존 전략입니다. 이유를 살펴보면: 1) **시장 검증이 우선**: 아무도 원하지 않는 완벽한 제품보다, 시장이 원하는 불완전한 제품이 낫습니다. 피봇 가능성을 고려하면 초기에 많은 시간을 아키텍처에 투자하는 것은 리스크입니다. 2) **속도가 생존**: 스타트업의 장점은 속도입니다. 기술 부채는 이 속도를 사는 대가입니다. 3) **자원 제약**: 적은 인력으로 많은 것을 해야 하므로 모든 것을 완벽하게 할 수 없습니다. 하지만 주의할 점도 있습니다: 1) "핵심 도메인"은 처음부터 잘 설계해야 합니다. 2) 보안, 개인정보 관련은 부채를 지면 안 됩니다. 3) 시리즈 A 이후에는 적극적으로 부채를 갚아야 합니다. 초기의 부채가 성장 단계에서 발목을 잡지 않도록 전환점을 명확히 인식해야 합니다.

**12. 인수인계 시 기술 부채에 대한 책임은 누구에게 있는가?**

> **답변**: 이 질문에는 두 가지 관점이 있습니다. **법적/공식적 책임**: 인수 후에는 새 팀이 책임집니다. 하지만 인수 과정에서 부채를 명확히 공개하지 않았다면 도의적 책임이 있습니다. **실무적 관점**: 1) **이전 팀의 책임** - 모든 기술 부채를 문서화하고 인수인계합니다. 영향도, 예상 비용, 히스토리를 포함합니다. 숨기면 안 됩니다. 2) **새 팀의 책임** - 인수 전 기술 실사(Technical Due Diligence)를 수행합니다. 코드베이스를 검토하고, 질문하고, 리스크를 평가합니다. 인수 후에는 현실을 받아들이고 관리 계획을 수립합니다. 3) **조직의 책임** - 인수 후 전환 기간(보통 3-6개월)을 제공해야 합니다. 이 기간에 기술 부채를 파악하고 정리할 시간을 줍니다. 핵심은 "투명성"입니다. 부채를 숨기고 인계하는 것은 비윤리적이며, 결국 조직 전체에 해가 됩니다.

## 8. 실제 회사 사례

### Google의 기술 부채 관리

Google은 "20% 프로젝트"로 유명하지만, 실제로는 기술 부채 관리를 위한 체계적인 시스템이 있습니다.

- **Readability 시스템**: 코드 품질을 유지하기 위해 언어별 Readability 인증을 도입. 인증받은 개발자만 해당 언어 코드를 승인할 수 있음
- **대규모 리팩토링 도구**: 수십억 줄의 코드베이스에서 안전하게 리팩토링할 수 있는 내부 도구 개발
- **Codebase Health 대시보드**: 각 팀의 코드 건강도를 실시간으로 모니터링
- **핵심 원칙**: "코드는 읽히는 것이 쓰이는 것보다 10배 많다" - 처음부터 읽기 좋은 코드를 작성

### Amazon의 기술 부채 접근법

Amazon은 "팀 자율성"과 "장기적 사고"를 기반으로 부채를 관리합니다.

- **Two-Pizza Team**: 작은 팀이 자신의 서비스를 완전히 소유. 부채도 팀이 책임짐
- **Bar Raiser 프로그램**: 채용 시 기술 수준을 높이 유지하여 부채 생성을 원천 차단
- **Working Backwards**: 고객 경험에서 시작해서 기술을 결정. "이것이 고객에게 어떤 영향을 미치는가?"로 부채 우선순위 결정
- **연간 OP1/OP2 계획**: 기술 부채 해결을 연간 계획에 포함

### Naver의 기술 부채 관리

국내 대표 IT 기업 Naver의 접근법입니다.

- **코드 리팩토링 데이**: 분기별로 1-2일을 리팩토링에 투자
- **기술 부채 백로그**: 각 팀별로 기술 부채를 공식 백로그로 관리
- **아키텍처 리뷰 위원회**: 주요 기술 결정과 부채 해결 방안을 검토하는 위원회 운영
- **신규 입사자 온보딩**: 코드베이스 히스토리와 기존 부채에 대한 문서 제공

### Kakao의 점진적 마이그레이션

Kakao는 급성장 과정에서 쌓인 기술 부채를 스트랭글러 패턴으로 해결했습니다.

- **모놀리스에서 MSA로**: 한 번에 전환하지 않고 트래픽이 적은 서비스부터 점진적으로 분리
- **기술 부채 가시화**: 전사 대시보드에서 각 팀의 기술 건강도를 공유
- **20% 규칙**: 스프린트의 20%를 기술 개선에 의무적으로 투자
- **테크 톡 문화**: 기술 부채 해결 경험을 공유하는 내부 발표 문화

### Toss의 기술 부채 철학

핀테크 스타트업 Toss는 속도와 품질 사이의 균형을 잘 유지합니다.

- **"완벽보다 완료"**: 초기에는 빠른 출시를 우선. 단, 핵심 도메인(결제, 보안)은 예외
- **기술 부채 예산**: 매 분기 기술 부채 해결에 할당할 예산(시간)을 미리 책정
- **자동화된 코드 품질 체크**: CI/CD 파이프라인에 정적 분석, 테스트 커버리지 체크 통합
- **리팩토링 페어링**: 레거시 코드 리팩토링 시 시니어-주니어 페어 프로그래밍으로 지식 공유

### Coupang의 대규모 마이그레이션 경험

빠르게 성장하며 대규모 기술 부채를 해결한 경험입니다.

- **SOA에서 MSA로**: 수년에 걸친 점진적 마이그레이션 프로젝트
- **Dark Launch**: 새 시스템을 기존 시스템과 병렬로 운영하며 검증
- **Feature Toggle**: 새 기능과 레거시 기능을 토글로 전환하며 안전하게 마이그레이션
- **블루-그린 배포**: 문제 발생 시 즉시 롤백 가능한 배포 전략

## 9. 팀 갈등 해결 방법

### 갈등 유형 1: 부채 해결 vs 신규 기능 우선순위

**상황**: 개발팀은 기술 부채 해결을 원하지만, PM은 신규 기능 개발을 요구

**해결 방법**:
1. **데이터로 대화**: "지난 3개월간 기능 개발 시간이 50% 증가했습니다. 이 추세라면 6개월 후에는 신규 기능 개발이 사실상 불가능합니다."
2. **비즈니스 언어 사용**: "지금 2주 투자하면 향후 분기에 2개 기능을 더 출시할 수 있습니다"
3. **타협점 찾기**: 신규 기능에 부채 해결을 "끼워넣기". "이 기능을 개발하면서 관련 레거시도 함께 정리하겠습니다"
4. **시간 확보 전략**: 버퍼 시간 활용, 스프린트 80% 규칙(20%는 기술 개선)

### 갈등 유형 2: 리팩토링 범위 논쟁

**상황**: 일부는 "작게 자주", 일부는 "한 번에 크게" 리팩토링을 주장

**해결 방법**:
1. **명확한 기준 수립**: 영향도와 위험도에 따라 접근법을 다르게
   - 소규모(1-2일): 보이스카우트 규칙
   - 중규모(1-2주): 스프린트 내 태스크
   - 대규모(1개월+): 스트랭글러 패턴
2. **파일럿 프로젝트**: 작은 범위에서 두 가지 접근을 실험하고 결과 비교
3. **위험 평가**: "빅뱅 접근의 위험은 무엇인가?" 토론으로 공감대 형성

### 갈등 유형 3: 누가 부채를 갚을 것인가

**상황**: 부채를 만든 사람이 해결해야 하는지, 현재 담당자가 해결해야 하는지

**해결 방법**:
1. **비난 없는 문화**: "누가 만들었는지"보다 "어떻게 해결할지"에 집중
2. **팀 책임**: 개인이 아닌 팀이 코드베이스를 소유한다는 원칙
3. **실용적 배분**: 해당 영역을 가장 잘 아는 사람이 담당 (만든 사람이 아닐 수 있음)
4. **지식 공유**: 리팩토링을 페어 프로그래밍으로 진행하여 지식 전파

### 갈등 유형 4: 부채 해결 품질 기준

**상황**: "이 정도면 됐다" vs "아직 부족하다"

**해결 방법**:
1. **사전에 완료 기준 정의**: 리팩토링 시작 전 Definition of Done 합의
2. **측정 가능한 목표**: "테스트 커버리지 80%", "사이클로매틱 복잡도 10 이하" 등
3. **단계별 목표**: 1차 목표(최소), 2차 목표(권장), 3차 목표(이상적) 설정
4. **시간 제한**: "2주 안에 할 수 있는 만큼만" 같은 현실적 제약 설정

## 10. 경영진/비개발자와의 소통 팁

### 효과적인 비유 모음

**집 비유**:
- "코드는 집과 같습니다. 급하게 방을 늘리고, 임시로 배관을 연결하다 보니 이제 새 방을 만들려면 기존 구조물을 피해 작업해야 합니다. 가끔 예전에 임시로 한 것들이 터지기도 합니다."

**공장 비유**:
- "우리 생산 라인(코드)의 기계들이 낡았습니다. 여전히 작동하지만, 고장이 잦고 새 제품 라인을 추가하기 어렵습니다. 기계를 교체하면 초기 비용이 들지만, 생산성이 크게 향상됩니다."

**신용카드 비유**:
- "기술 부채는 신용카드 빚과 같습니다. 당장은 편하지만 이자가 붙습니다. 우리는 매달 이자로 개발 시간의 30%를 쓰고 있습니다."

### 숫자로 말하기

```
[Before/After 비교]

- 버그 발생률: 주 2건 → 주 8건 (+300%)
- 온보딩 기간: 2주 → 4주 (+100%)

[비용 환산]
- 개발 시간 증가 비용: 월 200시간 × 10만원 = 2,000만원/월
- 버그 수정 비용: 건당 8시간 × 6건 × 10만원 = 480만원/월
- 총: 월 2,480만원, 연간 약 3억원
```

```markdown
### 조치하지 않을 경우
- 12개월 후 개발 속도 50% 추가 감소
- 기술 인재 이탈 가속화
- 경쟁력 상실
```

### 설득 타이밍

- **좋은 타이밍**: 신규 프로젝트 시작 전, 분기 계획 수립 시, 대형 장애 발생 후, 경쟁사가 앞서갈 때
- **나쁜 타이밍**: 급한 출시 직전, 조직 변경 중, 예산 삭감 시기
- **핵심 원칙**: 위기가 터지기 전에 미리 이야기하되, 데이터와 ROI로 무장

## 12. 체크리스트

### 기술 부채 식별 체크리스트

- [ ] 코드 리뷰에서 반복적으로 지적되는 사항을 수집했는가?
- [ ] 스프린트 회고에서 나온 기술 이슈를 정리했는가?
- [ ] 개발자들에게 "가장 고치고 싶은 것"을 물어봤는가?
- [ ] 정적 분석 도구(SonarQube 등)를 활용했는가?
- [ ] 온보딩 과정에서 신규 개발자의 혼란 지점을 파악했는가?
- [ ] 자주 버그가 발생하는 영역을 분석했는가?
- [ ] 오래된 TODO, FIXME, HACK 주석을 검색했는가?
- [ ] 구버전 라이브러리/의존성을 점검했는가?

### 기술 부채 기록 체크리스트

- [ ] 고유 ID를 부여했는가? (DEBT-XXX)
- [ ] 명확한 제목과 설명을 작성했는가?
- [ ] 영향받는 영역을 식별했는가?
- [ ] 비즈니스 영향을 분석했는가?
- [ ] 해결 비용을 추정했는가?
- [ ] 우선순위를 설정했는가?
- [ ] 담당자를 지정했는가?
- [ ] 백로그에 공식적으로 등록했는가?

### 기술 부채 해결 계획 체크리스트

- [ ] Definition of Done을 정의했는가?
- [ ] 단계별 마일스톤을 설정했는가?
- [ ] 위험 요소를 식별하고 완화 방안을 마련했는가?
- [ ] 필요한 테스트 전략을 수립했는가?
- [ ] 롤백 계획이 있는가?
- [ ] 이해관계자에게 일정을 공유했는가?
- [ ] 진행 상황 공유 방법을 정했는가?

### 이해관계자 설득 체크리스트

- [ ] 정량적 데이터를 준비했는가?
- [ ] 비즈니스 언어로 영향을 설명했는가?
- [ ] ROI를 계산했는가?
- [ ] 조치하지 않을 경우의 리스크를 설명했는가?
- [ ] 구체적인 해결 계획을 제시했는가?
- [ ] 적절한 타이밍에 제안하는가?
- [ ] 작은 것부터 시작해서 신뢰를 쌓는 전략이 있는가?

### 기술 부채 해결 후 체크리스트

- [ ] 효과를 측정했는가? (Before/After 비교)
- [ ] 교훈을 문서화했는가?
- [ ] 팀에 결과를 공유했는가?
- [ ] 이해관계자에게 보고했는가?
- [ ] 부채 목록에서 상태를 업데이트했는가?
- [ ] 비슷한 부채 발생을 방지하는 방안을 마련했는가?

### 테크리드 기술 부채 관리 역량 체크리스트

- [ ] 기술 부채를 체계적으로 식별하고 분류할 수 있다
- [ ] 영향도와 비용을 기반으로 우선순위를 정할 수 있다
- [ ] 비개발자에게 기술 부채의 영향을 설명할 수 있다
- [ ] ROI 기반으로 해결의 필요성을 설득할 수 있다
- [ ] 상황에 맞는 해결 전략(스트랭글러, 보이스카우트 등)을 선택할 수 있다
- [ ] 기술 부채 해결과 신규 기능 개발 사이의 균형을 잡을 수 있다
- [ ] 팀 내 부채 관련 갈등을 중재할 수 있다
- [ ] 정기적인 부채 회고를 주도할 수 있다
